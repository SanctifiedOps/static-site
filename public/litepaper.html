Static Litepaper

Ghost relay, real burns.

Static is a burn-to-prove, single-hop relay for encrypted transmissions on Solana. Messages are sealed as ephemeral NFT envelopes, routed through a blinded relay, and finalized by burning $STATIC so every delivery leaves a cost-proof but no breadcrumb to the sender. This document is the live, web-native litepaper linked from the app CTA; it expands on the mission, architecture, and economic primitives without asking you to download a PDF.

Mission

Build a network for operational crypto teams who need to move sensitive instructions with verifiable delivery, low latency, and zero residual metadata. Static keeps the payload encrypted end-to-end, preserves plausible deniability for the sender, and makes every successful transmission auditable through an on-chain burn receipt.

Problem

  • Cross-desk and cross-protocol coordination today relies on DM-style tooling that leaks metadata or on custom infra with no delivery proof.
  • Multi-hop onion routes add latency and surface area; broadcast channels create permanent trails.
  • Most “encrypted messaging” products stop at transport, not at verifiable delivery or cryptoeconomic skin-in-game.

Static in brief

Envelope

Encrypt payload with AES-GCM; store ciphertext behind a content hash; mint a disposable NFT that points to it.

Single blinded hop

Relay receives a sealed envelope, strips routing hints, forwards once. No routing tables, no onion state, minimal lag.

Burn-to-prove

Sender commits a burn of $STATIC within the same instruction set; burn hash is the auditable delivery receipt.

Recipient unlock

Recipient wallet detects the NFT, derives the shared secret (ECDH), decrypts, and self-destructs the envelope view.

Architecture

  • Program: Solana program (Anchor) that mints the envelope NFT, writes the payload URI hash, records burn amount, and emits MessageSent logs.
  • Relay: Stateless, single-hop forwarder. It only sees envelope headers and burn tier; it cannot read or re-route content.
  • Payload store: Short-lived, pinned object store. Only the encrypted blob is persisted; keys never leave sender/recipient devices.
  • Client: Web and CLI clients create an ephemeral keypair per send, encrypt locally, and push one transaction that mints + burns + dispatches.
  • Receipts: The burn is the receipt. Observers can verify “a transmission happened and was paid for” without learning who or what.

Token and burns

  • $STATIC: Utility asset for transmissions. Burns are mandatory; they create economic cost for spam and signal seriousness.
  • Tiers: Starter, Operator, Ghost. Higher tiers prioritize relay bandwidth and deeper obfuscation in queue ordering.
  • Distribution: A fraction of each burn routes to the relay operator pool; the rest is destroyed, tightening supply as usage grows.
  • Gas efficiency: Single transaction per send; no approval chatter; fits inside a Solana block comfortably for sub-second confirmation targets.

Security posture

  • Metadata minimization: No multi-hop gossip, no payload introspection, no persistent relay logs.
  • Forward secrecy: Ephemeral sender keys; recipients derive shared secrets per message; envelopes self-destruct in the client after open.
  • Abuse controls: Rate limits per sender wallet, minimum burn floor, and relay circuit-breakers tied to health metrics.
  • Audit: Third-party review in progress for program logic, burn accounting, and relay hardening.

Lifecycle of a transmission

  1. Wallet connects and stages a payload; user selects burn tier to set priority and cost.
  2. Client encrypts locally, uploads ciphertext to the sealed store, and generates the content hash.
  3. Program mints the envelope NFT, commits the hash, burns $STATIC, and emits MessageSent.
  4. Relay forwards once to the recipient endpoint; no acknowledgements are exposed publicly.
  5. Recipient wallet fetches ciphertext, derives the key, decrypts, and marks the envelope consumed.

Why single-hop, burn-backed

Static chooses a blinded single hop over onion chains because operational users care about latency and survivability. The burn enforces cost on every attempt, aligning the network around a simple truth: if you want to move in the dark, you pay to leave nothing behind. No breadcrumbs. Only a verifiable burn trail that proves the message was worth sending.
